理解JavaScript：原型

Aug 20, 2018 9719

为什么要去理解原型呢？这是因为尽管JavaScript是面向对象的，但是它并不是基于类的语言，而更像是基于原型的语言。每个对象都拥有一个原型对象，对象会以其原型为模版创建对象实例并从继承属性和方法。换句话说，理解原型是使用原型模式创建实例或者通过原型继承获得“父类”的属性和方法的前提。

什么是原型？

当我们创建一个函数时，这个函数会自动获得一个prototype（原型）属性。这个属性实际上是一个指针，它指向一个对象，这个对象就是函数的原型对象。哪怕不去使用这个属性，它也会一直存在。而在默认情况下，原型对象也会自动获得一个属性：constructor。它也是一个指针,它从原型对象指回prototype属性所在的函数，通过这个函数可以继续为原型对象添加更多的属性和方法。

当我们通过new关键字调用构造函数创建一个实例时，这个新对象会被执行[[ Prototype ]]链接，将该实例的内部属性指针[[prototype]]指向构造函数的原型对象。

由于这个指针的存在，实例可以访问原型对象中的属性。例如，实例中是不包含constructor属性的，但是可以通过指针访问原型对象中的constructor属性。

请注意这个连接并不存在于实例和其构造函数之间，而是存在于实例与构造函数的原型对象之间。

属性

属性读取

当代码读取某个属性时，会执行一次搜索。从对象实例本身开始，如果能够找到给定的属性，则直接返回值。否则搜索[[ Prototype ]]指针指向的原型对象，如果在原型对象中找到了属性，则返回该属性的值。

在读取foo1.bar的是时候，先后执行了两次搜索。第一次搜索实例本身，未能找到bar属性；第二次搜索[[ Prototype ]]指针指向的对象原型，找到了bar属性并返回值。在读取foo2.bar的时候也经历了同样的过程。多个实例就是这样共享原型对象中的属性和方法的。

属性设置

给对象设置属性的情况稍微有些复杂。在此我们暂时不考虑属性是只读的数据属性或者是访问器属性的情况。

当我们设置实例属性时，如果属性不是直接存在于实例中，则继续搜索[[ Prototype ]]指针指向的原型对象。如果在原型对象上找不到给定属性，则直接将其添加到实例上；如果给定属性存在于原型对象上，则会给实例添加一个新属性，这个属性会屏蔽原型对象上的同名属性，阻止我们对它的访问。

属性继承

在MDN上是如此概括属性继承的工作原理的：

JavaScript 对象是动态的属性“包”（指其自己的属性）。JavaScript 对象有一个指向一个原型对象的链。当试图访问一个对象的属性时，它不仅仅在该对象上搜寻，还会搜寻该对象的原型，以及该对象的原型的原型，依次层层向上搜索，直到找到一个名字匹配的属性或到达原型链的末尾。

我们在属性读取的例子里，实际上就用到了属性的继承，实例本身不包含给定的属性，它们返回的属性的值是从[[ Prototype ]]指针指向的原型对象上获得的。

这不仅对属性有效，对方法同样适用。我们能实现这些的原因就是JavaScript基于原型链的继承。

在深入讨论继承之前，我们现在此深入了解以下原型链吧！

原型链

我们一开始提到的当我们定义一个将被函数时，会自动生成一个property属性。这个属性除了用来包含函数被“构造调用”而创建的实例共享的属性和方法外，还有一个作用，那就是实现基于原型的继承。

我们已经知道在调用构造函数创建实例时，实例会包含一个[[ Prototype ]]指针，指向构造函数的原型对象。如果我们更进一步，让一个原型对象等于另一个构造函数创建的实例的话，这个原型对象也会拥有一个指向另一个构造函数的原型对象的[[ Prototype ]]指针。这种做法对另一个原型对象同样适用，当我们层层递进连接起不同的原型对象后，就构成了实例与原型对象的链条，这一系列对象的链接就是原型链。

接下来我们详细的分析一下原型链形成的过程，并掌握其中的关键。

详细步骤分析

步骤1. 创建一个Parent实例

一个Parent的实例应该能够访问以下属性：实例自身包含的value属性，一个[[ Prototype ]]指针指向Parent的原型对象，通过这个指针，它可以访问getValue函数和constructor指针。

步骤2. 用Parent实例取代默认Child.prototype

在重写Child.prototype以后，它也拥有了一个Parent实例该有的一切。唯一的不同是，value本来是在Parent构造函数里定义的实例（非共享）属性，而现在则成了Child.prototype上的原型（共享）属性。

步骤3. Child的实例从Parent.prototype和新Child.prototype中继承属性和方法

这是否让你回想起了我们在属性读取部分使用的例子？在这里instance.getValue()为了获得属性和方法经历了三个步骤，搜索instance；搜索Child.prototype；搜索Parent.prototype。直到最后一步才找到需要的属性和方法。在找不到属性或方法的情况下，搜索过程总是要一环一环的前行到原型末端才会停下来。

此外，在原型链中出现的原型都可以是原型链所派生的实例的原型。因此，我们在例子的最后验证了Parent.prototype和Child.prototype都是instance的原型。

步骤4. 在继承的基础上为Child构造函数和原型添加新属性和方法

在建立好了原型链以后，现在我们可以考虑在Child构造函数中添加实例属性或者在Child.prototype上添加共享属性或者方法了。

这里需要注意的是，当我们在Child的构造函数里定义了value属性后，实例instance不再访问Child.prototype中的value属性（值为“Parent”），而是直接返回自身包含的属性值“Child”。这就是我们之前提到过的“屏蔽”。

但是原型链存在两个问题。第一个问题我们已经体验到了，那就是当我们将Parent类的实例赋给Child.prototype后，原先的实例属性变成了原型属性，本来希望实例自己保存的属性，变成了共享属性，这肯定不是我们想要的。第二个问题是：在创建Child类型的实例时，不能向Parent的构造函数中传递参数。
